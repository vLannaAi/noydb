# NOYDB v0.6.0 — Query DSL completion + `.noydb` container

> **Status:** draft, not yet shipped. Final notes will be regenerated by the
> changeset CLI on `pnpm changeset version` after the 8 v0.6 PRs land.
> **Theme:** finish the query DSL story so consumers can express joins and
> aggregations directly in `.query()` / `.scan()` instead of folding in
> userland, and ship the `.noydb` container format that v0.10 + v0.11 will
> build on.

## Highlights

- **`.join()`** — eager single-FK joins, multi-FK chaining, and live mode.
  Resolves `ref()`-declared foreign keys into attached right-side records,
  with three planner strategies and three ref-mode behaviors on dangling refs.
- **`.live()`** — reactive primitive on `Query<T>` with merged change streams
  across every join target. Frame-agnostic; Vue / React / Solid wrappers can
  layer on top.
- **`.aggregate()`** — five built-in reducers (`count`, `sum`, `avg`,
  `min`, `max`) with a chainable `.aggregate({...}).run()` synchronous
  terminal and a `.aggregate({...}).live()` reactive terminal.
- **`.groupBy()`** — single-level partitioning before `.aggregate()` with
  cardinality warning at 10k groups and a hard `GroupCardinalityError`
  at 100k.
- **`scan().aggregate()`** — memory-bounded streaming aggregation. O(reducers)
  state, not O(records). Same reducer protocol as `Query.aggregate()` —
  reducers plug into either path unchanged.
- **`scan().join()`** — streaming joins over `scan()`. Bypasses the eager
  join's 50,000-row ceiling for collections that exceed it.
- **`.noydb` container format** — minimum-disclosure binary wrapper around
  `compartment.dump()` for safe storage in cloud folders. Magic-byte
  prefix, opaque ULID handle, integrity hash, brotli/gzip compression.

**Stats:** 8 features closed, **142 new tests**, **~6,400 lines added**
across `@noy-db/core` and `@noy-db/file`. Zero new dependencies — every
feature respects the zero-runtime-deps invariant.

## Joins (#73 / #75 / #74 / #76)

The eager join is the foundation: chain `.join(field, { as })` on any
`Query` built from `collection.query()` to resolve a `ref()`-declared
foreign key.

```ts
const rows = invoices.query()
  .where('status', '==', 'open')
  .join<'client', Client>('clientId', { as: 'client' })
  .toArray()
// → [{ id, amount, status, client: { id, name, tier } }, ...]
```

### Eager joins (#73)

- **Two planner strategies**, auto-selected:
  - **Indexed nested-loop** when the right source exposes `lookupById`
    (the common case for Collection right sides) — O(1) per left row
  - **Hash join** otherwise — build `Map<id, record>` once from the
    right side's `snapshot()`, probe per left row
- **Hard row ceiling** at 50,000 per side (`JoinTooLargeError`),
  overridable via `{ maxRows }`. One-shot warning at 80% of the
  ceiling on the existing warn channel.
- **Three ref-mode behaviors** on dangling refs:
  - `strict` → throws `DanglingReferenceError`
  - `warn` → attaches `null` + one-shot warning per unique pair
  - `cascade` → attaches `null` silently
- **Same-compartment only** by construction. Cross-compartment
  correlation continues to go through `Noydb.queryAcross` (#63 from
  v0.5).

### Multi-FK chaining (#75)

```ts
invoices.query()
  .join<'client', Client>('clientId', { as: 'client' })
  .join<'category', Category>('categoryId', { as: 'category' })
  .toArray()
// → [{ ..., client: {...}, category: {...} }, ...]
```

Each leg picks its own strategy and ref mode independently. Per-leg
`maxRows` is enforced against the current left-row count, so a query
like `.join('a', { maxRows: 100_000 }).join('b', { maxRows: 50 })`
correctly fires the second-leg ceiling on a 60-row left set.

### Live mode (#74)

```ts
const live = invoices.query()
  .where('status', '==', 'open')
  .join<'client', Client>('clientId', { as: 'client' })
  .live()

live.subscribe(() => render(live.value))
// later
live.stop()
```

`.live()` returns a `LiveQuery<T>` — a frame-agnostic reactive primitive
with `value` / `error` fields and a `subscribe(cb)` notification channel.

For joined queries it additionally subscribes to **every join target's
change stream**, so right-side mutations re-fire the live query. Targets
are deduped by collection name, so a chain joining the same target twice
only subscribes once.

If a re-run throws (e.g. strict ref-mode encounters a dangling ref
mid-flight), the previous successful value is preserved and the error is
stored in `live.error` — the throw does not propagate out of the source's
change handler, which would tear down the upstream emitter.

### Streaming joins (#76)

```ts
for await (const inv of invoices.scan().join<'client', Client>('clientId', { as: 'client' })) {
  await processInvoice(inv) // inv.client is attached
}
```

The eager join has a 50,000-row ceiling on the left side. For collections
that exceed it, `scan().join()` streams the left side page-by-page from
the adapter — memory ceiling on the left becomes O(pageSize), not
O(rowCount). The right side is held in memory either via per-row
`lookupById` (the indexed path, O(1) per row) or as a hash built once
from `snapshot()` (the fallback path, same as eager hash strategy).

**True left-and-right streaming joins** (sort-merge planner) are out of
scope for v0.6 — file an issue if a real consumer needs it.

## Aggregations v1 (#97 / #98 / #99)

The aggregation primitives are designed around a single `Reducer<R, S>`
protocol that plugs into both `Query.aggregate()` (in-memory) and
`Scan.aggregate()` (streaming) without duplication.

### Reducers + `.aggregate()` (#97)

```ts
import { count, sum, avg, min, max } from '@noy-db/core'

// Synchronous, in-memory
const { total, n, mean } = invoices.query()
  .where('status', '==', 'open')
  .aggregate({
    total: sum('amount'),
    n:     count(),
    mean:  avg('amount'),
  })
  .run()

// Reactive
const live = invoices.query()
  .where('status', '==', 'open')
  .aggregate({ total: sum('amount'), n: count() })
  .live()
live.subscribe(() => updateDashboard(live.value))
```

- **Five built-in reducers:** `count()`, `sum(field)`, `avg(field)`,
  `min(field)`, `max(field)`
- **Empty result sets** return `null` (not NaN, not ±Infinity) for
  `avg` / `min` / `max` — avoids poisoning downstream arithmetic
- **Live mode** uses naive full re-run on source change. Incremental
  delta maintenance is admitted by the reducer protocol's `remove()`
  hook but not driven by the v0.6 executor — a future PR can switch
  without an API break.
- **Type inference** flows through the spec: a spec of
  `{ total: sum('amount'), n: count() }` infers a result of
  `{ total: number, n: number }`.

### `.groupBy()` (#98)

```ts
const byClient = invoices.query()
  .where('status', '==', 'open')
  .groupBy('clientId')
  .aggregate({ total: sum('amount'), n: count() })
  .run()
// → [
//     { clientId: 'c1', total: 5250, n: 3 },
//     { clientId: 'c2', total: 1200, n: 1 },
//   ]
```

- **Insertion-order emission** via JS `Map` — first-seen keys come
  first. Consumers who want a specific ordering should `.sort()`
  downstream.
- **Cardinality caps:** one-shot warning at 10,000 distinct groups,
  hard `GroupCardinalityError` at 100,000. Grouping on a high-uniqueness
  field like `id` is almost always a query mistake; the error message
  names the offending field and observed cardinality.
- **`null` and `undefined` keys are kept in separate buckets** —
  `Map` distinguishes them and merging would hide a real distinction
  between "field explicitly set to null" and "field not set at all."
- **Type-level `dictKey` narrowing prep** — v0.8 will add an overload
  that narrows the group key to a stable dictionary key, preventing
  the silent bug where grouping by a localized label produces
  different buckets per reader.

### `scan().aggregate()` (#99)

```ts
const { total, n } = await invoices.scan({ pageSize: 1000 })
  .where('year', '==', 2025)
  .aggregate({ total: sum('amount'), n: count() })
```

- **Memory: O(reducers), not O(records).** The aggregator initializes
  one mutable state per reducer, iterates the stream one record at a
  time via `for await`, and never collects records into an array.
  This is what makes streaming aggregation suitable for collections
  that exceed the in-memory query ceiling.
- **Returns a `Promise`** (unlike `Query.aggregate().run()` which is
  synchronous) — the scan is inherently async because it walks
  adapter pages.
- **No `.live()` in v0.6.** Reconciling unbounded streaming with a
  change-stream subscription is a design problem, not just a code
  one. Consumers with huge collections and live needs should narrow
  with `.where()` enough to fit in `query()` and use
  `query().aggregate().live()` instead.

### Reducer reuse — the load-bearing design

Every factory accepts an optional `{ seed }` parameter plumbed
through the protocol but unused by the v0.6 executor. This is **#87
constraint #2** — the load-bearing seam for v0.10 partition-aware
aggregation. When v0.10 lands, the seed will carry running state
across partition boundaries without requiring an API break.

The reducer protocol — `{ init, step, remove?, finalize }` — was
deliberately designed so both `Query.aggregate()` and
`Scan.aggregate()` could share it. No duplicated executors, no
duplicated factory exports.

## `.noydb` container format (#100)

`compartment.dump()` already produces a JSON string with encrypted
records inside. The new `.noydb` container is a thin minimum-disclosure
wrapper that makes the file safe to drop into cloud storage (Drive,
Dropbox, iCloud) without leaking compartment names or exporter identity
through the cloud's metadata API.

```ts
import { writeNoydbBundle, readNoydbBundle } from '@noy-db/core'
import { saveBundle, loadBundle } from '@noy-db/file'

// Core primitives
const bytes = await writeNoydbBundle(company)
const { header, dumpJson } = await readNoydbBundle(bytes)

// File adapter helpers
const handle = await company.getBundleHandle()
await saveBundle(`./bundles/${handle}.noydb`, company)
const result = await loadBundle(`./bundles/${handle}.noydb`)
```

### Format byte layout

```
+--------+--------+--------+--------+
|  N     |  D     |  B     |  1     |  Magic 'NDB1' (4 bytes)
+--------+--------+--------+--------+
| flags  | compr  |  header_length (uint32 BE)            |
+--------+--------+--------+--------+--------+--------+--------+
| header_length bytes of UTF-8 JSON header                       ...
+--------+--------+
| compressed body bytes                                            ...
```

10-byte fixed prefix, then variable header JSON, then the compressed
body.

### Minimum-disclosure header

**Only four allowed keys**, validated at parse time:

```json
{
  "formatVersion": 1,
  "handle": "01HYABCDEFGHJKMNPQRSTVWXYZ",
  "bodyBytes": 41234567,
  "bodySha256": "abc123..."
}
```

Forbidden (every key produces a parse error): `compartment`,
`exporter`, `_exported_at`, KDF params, salt fields, anything starting
with underscore. The validator allowlist is closed — forward-compat
extension keys require a format version bump and a new validator.

### Stable opaque handles via ULID

`compartment.getBundleHandle()` generates and persists a 26-character
Crockford base32 ULID on first call to a reserved `_meta/handle`
envelope, returns the same handle on subsequent calls. Survives process
restarts (the envelope is on the adapter, not just in memory). Different
compartments get different handles; the same compartment exporting
multiple times always gets the same handle.

The ULID is the only identifier in the bundle header — opaque, doesn't
leak compartment names, and is the planned primary key for v0.11 cloud
bundle adapters.

### Compression

- **Brotli** when `CompressionStream('br')` is supported (Node 22+,
  Chrome 124+, Firefox 122+) — typically 30-50% smaller than gzip on
  JSON payloads
- **Gzip** fallback (always available in Node 18+)
- **None** option for round-trip testing or piping into a separately
  compressed transport

The writer feature-detects brotli at runtime and falls back to gzip
silently when `{ compression: 'auto' }` (the default) is passed.
Explicit `{ compression: 'brotli' }` throws on unsupported runtimes.

### Integrity verification

`bodyBytes` and `bodySha256` describe the **compressed** body bytes
(not the decompressed dump). This lets `readNoydbBundleHeader()` verify
integrity without decompressing — useful for fast cloud-side validation
(an edge worker can verify a bundle's hash from the first ~200 bytes,
no decompression cost).

Tampering with any byte of the body produces `BundleIntegrityError`.
A length mismatch fires before the SHA check (cheaper, more actionable
error message).

## Partition-awareness seams (#87)

#87 is a meta-issue: design constraints that v0.6 work must honor so
v0.10 partitioned collections don't require an API break. **Zero
partition code ships in v0.6**; the seams just preserve optionality.

| Constraint | Honored where |
|---|---|
| `JoinLeg.partitionScope: 'all' \| string[]` plumbed through the planner | #73 (eager join), #76 (streaming join) — every JoinLeg constructed in v0.6 sets `partitionScope: 'all'` |
| Reducer factories accept `{ seed }` parameter | #97 (every factory: `count`/`sum`/`avg`/`min`/`max`), #98 (delegates to reducers), #99 (delegates to reducers) |
| Export envelope reserves `auxiliaryState` field for partition carries | (deferred — gated on a real v0.8 i18n consumer ask) |

A dedicated test in `query-aggregate.test.ts` pins the seed seam:
identical results with and without seeds, proving the parameter is
plumbed but unused at execution time. Removing the parameter would
break the seam silently — that's why the factories reference it via
`void _seed` instead of dropping it entirely.

## Migration & breaking changes

**One backward-compat-affecting change** in v0.6:

### `Collection.scan()` return type

The return type of `Collection.scan()` changed from
`AsyncIterableIterator<T>` to `ScanBuilder<T>`. **Every existing**
`for await (const rec of collection.scan()) { … }` call continues to
work unchanged — `ScanBuilder` implements `[Symbol.asyncIterator]` and
is fully drop-in for the iteration use case.

**What no longer works:** direct `.next()` / `.return()` / `.throw()`
calls on the scan result. A pre-change grep confirmed zero call sites
in the repo or any first-party consumer use this pattern, but if you
have userland code that does, the migration is to wrap in a normal
`for await` loop.

All 36 existing `pagination` and `lazy-hydration` tests pass without
modification, which is the main backward-compat proof.

## What didn't make it (out of scope, tracked separately)

These are deliberately not in v0.6. Each one has a documented reason and
a planned home in a later release.

| Feature | Why deferred |
|---|---|
| Aggregations across joins | No real consumer ask; would require executor-level coupling between the join and aggregate paths |
| Per-row callback reducers (`.reduce(fn, init)`) | The factory protocol covers every common case; callback reducers add a foot-gun for non-associative folds |
| Index-backed aggregation planner | Optimization, not a feature; gated on a real consumer ask |
| Multi-level `groupBy` (nested groupings) | Most consumers want one level; multi-level is a substantially different planner |
| `.having(predicate)` filtering on grouped results | Composes with userland `.filter()`; saves a method on the surface |
| `scan().aggregate().live()` | Reconciling unbounded streaming with change streams is a design problem, not a code one |
| `scan().groupBy().aggregate()` | High-cardinality grouping on huge collections re-introduces the O(groups) memory problem that streaming aggregate was designed to avoid |
| Per-bucket incremental delta maintenance for live aggregation | v2 optimization; the reducer protocol already admits it |
| True left-and-right streaming joins (sort-merge) | No consumer ask; current streaming joins still hold the right side in memory |
| Filtering on joined fields after `.join()` | Would require swapping pipeline ordering; clauses currently run before joins |
| LRU + lazy probe for non-`lookupById` right sources in streaming joins | The current hash-from-snapshot fallback materializes the right side once; LRU only matters when the right side itself is stream-only, which v0.6's adapter API doesn't model |
| `scan().join().live()` | Same design problem as `scan().aggregate().live()` |
| Bundle adapter shape (Drive, Dropbox, iCloud) | v0.11 #93 — depends on `.noydb` format landing first (this release) |
| `noydb inspect/open` CLI commands | v0.10 — depends on `.noydb` format landing first (this release) |
| Browser extension reader | v0.10 |
| Multi-compartment bundles | v2 — single-compartment is the expected unit for v0.6 |
| Streaming decompression for mobile | v2 — current full-buffer decompression is fine for the <50MB bundle sizes v0.6 targets |
| ZIP-like selective extraction | v2 |
| Encrypting the dump body itself | Would require a second key derivation; bigger design conversation than v0.6 hosts |
