# v0.12 Blob Store — Design Document

> **Issues:** #103 (`NoydbBundleStore`) · #105 (encrypted binary blob store)
> **Status:** design approved, revision 2 — post-review enhancements
> **Last updated:** 2026-04-10

---

## Background

v0.12's storage milestone introduces first-class binary data alongside the existing
encrypted record store. Two issues drive this work:

| Issue | Title | Scope |
|------:|-------|-------|
| #103 | `NoydbBundleStore` interface | Second store shape for blob-backend adapters (Drive, WebDAV, iCloud) |
| #105 | Encrypted binary blob store | Blobs alongside records — content-addressed, deduplicated, versioned |

Both share the same underlying primitives (`BlobObject`, ref counting, chunk layout)
but address different storage topologies.

---

## Use cases

### UC-1 — Generated files attached to a record

An invoice record generates one or more PDFs (e.g. localised copies: `invoice_en`,
`invoice_th`, `receipt`). These are not independent records — they are derived outputs
of an existing record and are accessed through it.

- Multiple named slots per record.
- Each slot holds one **current** binary.
- The binary can be streamed for browser preview (`Content-Disposition: inline`) or
  served as a download (`Content-Disposition: attachment`).
- When the invoice changes, the PDF is regenerated and replaces the current slot.
  The old version is discarded **unless** it was explicitly published (see UC-3).

### UC-2 — File as the record (blob-primary collection)

A "scanned invoices" collection where the uploaded file **is** the primary content
of the record. The record's structured metadata (upload date, page count, etc.) is
secondary. The blob is not an attachment — it is the point of the record.

- Single blob per record by convention (`primary` slot).
- Standard `collection.put(id, metadata)` + `collection.blob(id).put('primary', bytes)`.
- No special API required; the `BlobSet` handle covers this case.
- A future `collection.putBlob(id, bytes, { meta })` shorthand may be added in v0.13
  if the pattern is common enough.

### UC-3 — Amendment versioning (selective history)

An issued invoice is locked. When an amendment is required:

1. The current PDF was already published as `'issued-2025-01'`.
2. The invoice record is amended → new PDF generated.
3. New PDF overwrites the current slot.
4. New PDF is published as `'amendment-2025-02'`.

Result: both PDFs are retained (each holds an independent eTag reference). The slot
always reflects the latest (amended) version; named published versions give access
to any explicitly labelled snapshot.

noy-db history tracks **record** changes (field edits). Blob history is
**independent** — only explicitly published blobs are retained, not every
byte-level change. This keeps storage predictable.

### UC-4 — Cross-record deduplication

The same PDF template (e.g. a company letterhead) is embedded in hundreds of
invoices. Uploading it repeatedly should not multiply storage.

Content-addressed storage ensures identical bytes are stored once. Multiple
slots across multiple records reference the same `eTag`. Reference counting
tracks how many references exist so the underlying data can be safely GC'd
when no references remain.

#### eTag computation — keyed hash (HMAC-SHA-256)

A plain `SHA-256(plaintext)` hash leaks content equality to the store: an
attacker with store access can pre-compute the hash of a known document (e.g.
a standard template) and confirm which records have it attached, even without
decryption keys. This undermines the zero-knowledge guarantee.

Instead, the eTag is computed as:

```
eTag = HMAC-SHA-256(key = _blob DEK raw bytes, message = plaintext bytes)
```

This preserves deduplication within a vault (same key, same plaintext → same
eTag) while making the eTag opaque to the store. An attacker without the
`_blob` DEK cannot reverse or predict eTag values.

**Trade-off:** deduplication is vault-scoped, not global. Two vaults with
identical files will store separate copies. This is acceptable — cross-vault
dedup would require a shared key, which contradicts vault isolation.

**Key rotation impact:** rotating the `_blob` DEK changes all eTags. A
rotation must re-compute eTags and update all slot references. This is an
O(blobs) migration, but `_blob` DEK rotation is rare (only on user revocation
from the vault-level blob key).

### UC-5 — S3 and other large-object backends

DynamoDB has a 400 KB item limit. S3 has no practical object size limit.
IndexedDB and the file store are similarly unconstrained.

The chunk loop (256 KB per KV record, fitting within DynamoDB) should only be
applied when the backing store requires it. S3 and file stores should store each
blob as a single object — chunking adds overhead with no benefit and works against
S3's native multipart upload.

### UC-6 — Future partition-aware execution (#87 seams)

A blob with a given `eTag` may eventually be shared across two partitions within
the same vault. Reference counting must span partitions to avoid premature GC.

The `BlobObject.refCount` field is designed as a scalar for v0.12 (vault-wide).
In v0.13+ it will be extended to a partition-keyed map without a format break:

```
// v0.12
refCount: number

// v0.13+
refs: { vault: number; [partitionId: string]: number }
```

GC logic uses `refCount` (or the sum of `refs` values) — the change is internal.

### UC-7 — Chunk integrity against reorder/substitution

Individual AES-256-GCM encryption per chunk provides per-chunk authentication
but does not bind chunks to each other or to their parent blob. Without an
integrity chain, an attacker with store write access could:

1. **Reorder chunks** — swap chunk 0 and chunk 1. Each decrypts fine
   individually but reassembly produces corrupted output.
2. **Substitute chunks** — replace chunk 2 of blob A with chunk 2 of blob B
   (same index, different eTag). Decryption succeeds if both use the same DEK.
3. **Truncate** — remove the last chunk. The reader computes a lower chunk
   count from `compressedSize` and silently produces truncated output.

Mitigation: include `eTag` and `chunkIndex` in the AES-GCM Additional
Authenticated Data (AAD) field of every chunk encrypt call. Store `chunkCount`
in `BlobObject` so the reader knows exactly how many chunks to expect. See
"Chunk integrity (AAD binding)" section below.

### #103 — `NoydbBundleStore`

A second store shape for backends that operate on whole-vault bundles rather than
per-record KV. The `kind: 'bundle'` discriminant lets the engine auto-detect the
store shape. Backends implement five methods:

```ts
interface NoydbBundleStore {
  readonly kind: 'bundle'
  readonly name?: string

  /** Read the entire vault. Returns null for a brand-new vault. */
  readBundle(vaultId: string): Promise<{ bytes: Uint8Array; version: string } | null>

  /**
   * Write the entire vault. `expectedVersion` is the version token from the
   * last `readBundle` (or null for a first write). Implementations MUST reject
   * the write if the stored version has advanced past `expectedVersion`
   * (optimistic concurrency control). On success, returns the new version.
   */
  writeBundle(
    vaultId: string,
    bytes: Uint8Array,
    expectedVersion: string | null,
  ): Promise<{ version: string }>

  /** Delete a vault bundle. Idempotent — no-op if the bundle does not exist. */
  deleteBundle(vaultId: string): Promise<void>

  /** List all vault bundles managed by this store. */
  listBundles(): Promise<Array<{ vaultId: string; version: string; size: number }>>
}
```

#### Optimistic concurrency at the bundle layer

Two browser tabs (or devices) writing to the same Google Drive file would
silently overwrite each other without OCC. The `expectedVersion` token on
`writeBundle` prevents this:

1. On `open()`: `readBundle()` returns `{ bytes, version: 'v3' }`.
2. Mutations accumulate in the in-memory snapshot.
3. On flush: `writeBundle(id, bytes, 'v3')` → backend checks that the stored
   version is still `'v3'`. If another writer advanced it to `'v4'`, the call
   throws `BundleVersionConflictError` and the engine can pull-merge-retry.

Backend implementations map `version` to their native concurrency primitive:
- Google Drive: `If-Match` ETag header
- WebDAV: `If-Match` / `Lock-Token`
- S3: `If-Match` ETag
- Filesystem: mtime or content-hash

#### `wrapBundleStore`

`wrapBundleStore(bundle: NoydbBundleStore): NoydbStore` converts a bundle backend
into the standard six-method `NoydbStore`. The wrapper holds a per-vault in-memory
`VaultSnapshot`, lazy-loaded on first access.

**Flush modes:**

By default the wrapper flushes on every mutation — O(vault size) per write. For
bulk operations, callers should use one of:

1. **`syncPolicy` debounce** (recommended): pair with
   `syncPolicy: { push: { mode: 'debounce', ms: 500 } }` from #101.
2. **Explicit batching**: `wrapBundleStore(bundle, { autoFlush: false })` disables
   auto-flush. The caller must invoke `store.flush(vaultId)` to persist:

```ts
const store = wrapBundleStore(driveStore({ folderId: '...' }), { autoFlush: false })
const db = await createNoydb({ store, ... })
// ... bulk mutations ...
await store.flush(VAULT)   // single network write
```

3. **Transaction helper**: `store.batch(vaultId, async (snap) => { ... })` defers
   flushing until the callback completes.

Wire format: `JSON.stringify({ _noydb_bundle_store: 1, vault, ts, data: VaultSnapshot })`.
No additional encryption — the envelopes inside are already AES-GCM ciphertext.

The wrapper tracks the bundle `version` token from the last read and passes it as
`expectedVersion` on every flush. On `BundleVersionConflictError`, the wrapper
re-reads, merges the remote snapshot into the local one (last-write-wins per
record key), and retries the flush (max 3 attempts).

Concrete `to-*` adapters (`@noy-db/to-drive`, `@noy-db/to-webdav`) ship in v0.13.

### #105 — Encrypted binary blob store

#### Storage layout

```
_blob_index / {eTag}
    BlobObject — immutable once written
    DEK: _blob  (vault-shared across all collections → enables UC-4 deduplication)

_blob_chunks / {eTag} / {chunkIndex}
    Encrypted chunk bytes (AES-256-GCM, fresh IV per chunk)
    DEK: _blob
    Only used for KV stores with item-size limits (DynamoDB).
    S3, file, and IDB stores: single record per eTag (chunkIndex always 0).

_blob_slots_{collection} / {recordId} / {slotName}
    SlotRecord — current slot pointer, versioned
    DEK: parent collection  (ACL boundary — protects which blobs are on which records)

_blob_versions_{collection} / {recordId} / {slotName} / {label}
    VersionRecord — explicitly published snapshot
    DEK: parent collection
```

The split between `_blob_index` (vault `_blob` DEK) and `_blob_slots_*` (parent
collection DEK) is the key ACL design: the blob data is shared but who can discover
which blob belongs to which record is gated by the collection's DEK.

#### BlobObject

```ts
interface BlobObject {
  eTag: string            // HMAC-SHA-256(blob DEK, plaintext) — opaque to store
  size: number            // original uncompressed bytes
  compressedSize: number  // bytes actually encrypted and chunked
  compression: 'gzip' | 'none'
  chunkSize: number       // raw chunk size used at write time (bytes)
  chunkCount: number      // total chunks written — reader MUST verify this
  mimeType?: string
  createdAt: string
  refCount: number        // live reference count (slots + published versions)
}
```

#### Chunking and the `maxBlobBytes` capability hint

`StoreCapabilities` gains a new optional field:

```ts
interface StoreCapabilities {
  casAtomic: boolean
  auth: StoreAuth
  maxBlobBytes?: number   // undefined = no limit; 256*1024 for DynamoDB
}
```

Chunk size selection priority:

1. Explicit `opts.chunkSize` passed to `blob.put()` (caller override).
2. `store.capabilities.maxBlobBytes` if declared by the store.
3. `DEFAULT_CHUNK_SIZE` (256 KB) — safe fallback for DynamoDB.

When the effective chunk size covers the entire compressed payload, the blob is
stored as a single `_blob_chunks/{eTag}/0` record (no loop). Otherwise it is
split into `Math.ceil(compressedSize / chunkSize)` chunks.

The effective `chunkSize` and resulting `chunkCount` are recorded in `BlobObject`
at write time. Readers MUST use `blob.chunkSize` and `blob.chunkCount` — never
recompute from `compressedSize / DEFAULT_CHUNK_SIZE`, as the write may have used
a different chunk size.

**`maxBlobBytes` per built-in store (v0.12 target):**

| Store | `maxBlobBytes` |
|-------|---------------|
| `to-memory` | `undefined` (no limit) |
| `to-file` | `undefined` |
| `to-browser-local` | `5 * 1024 * 1024` (5 MB localStorage quota safety) |
| `to-browser-idb` | `undefined` |
| `to-aws-dynamo` | `256 * 1024` (400 KB item limit minus envelope overhead) |
| `to-aws-s3` | `undefined` |

#### Chunk integrity (AAD binding)

Each chunk is encrypted with AES-256-GCM using Additional Authenticated Data
(AAD) that binds the chunk to its parent blob and position:

```ts
// AAD = UTF-8 encoding of "{eTag}:{chunkIndex}:{chunkCount}"
const aad = new TextEncoder().encode(`${eTag}:${chunkIndex}:${chunkCount}`)
```

On decrypt, the reader reconstructs the same AAD from the `BlobObject` metadata
and passes it to `crypto.subtle.decrypt()`. If any chunk has been reordered,
substituted from a different blob, or if the total chunk count was tampered with,
the AES-GCM auth tag verification fails and throws `TamperedError`.

This provides:
- **Anti-reorder:** chunk 0 decrypted with `aad="...eTag:1:4"` fails.
- **Anti-substitution:** chunk from blob B decrypted with blob A's eTag in AAD fails.
- **Anti-truncation:** reader expects exactly `chunkCount` chunks; fewer = error.

#### Write ordering (safe partial failure)

1. Compute `eTag = HMAC-SHA-256(blobDEK, plaintext)`.
2. Auto-detect MIME type from magic bytes if not provided (see "MIME type detection").
3. Compress plaintext (gzip, unless `compress: false` or already-compressed MIME type).
4. Determine effective chunk size (see "Chunking" priority above).
5. Write **chunks first** (before index entry), each with AAD binding.
   A partial failure here leaves orphan chunks — harmless, GC-able. No broken
   index entry pointing at missing chunks.
6. Write `BlobObject` to `_blob_index` with `refCount: 1` and `chunkCount`/`chunkSize`.
7. If the eTag already existed (deduplication path): CAS-increment `refCount`
   via retry loop (see below). Skip chunk writes — they already exist.
8. Write `SlotRecord` to `_blob_slots_{collection}` via CAS retry loop.
   If overwriting an existing slot, the old eTag's `refCount` is decremented
   in the same retry pass.

#### Reference counting (CAS-based)

Every reference change uses an optimistic CAS retry loop:

```ts
async function casUpdateRefCount(
  store: NoydbStore, vault: string, eTag: string,
  delta: number, getDEK: () => Promise<CryptoKey>,
  maxRetries = 5,
): Promise<void> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const { blob, version } = await loadBlobObjectVersioned(store, vault, eTag, getDEK)
    if (!blob) throw new NotFoundError(`BlobObject ${eTag} not found`)
    const updated = { ...blob, refCount: blob.refCount + delta }
    try {
      await writeBlobObject(store, vault, updated, version)
      return
    } catch (err) {
      if (err instanceof ConflictError && attempt < maxRetries - 1) continue
      throw err
    }
  }
}
```

| Event | `refCount` change |
|---|---|
| New blob written (first put) | Created at 1 |
| Deduplication (existing eTag, new slot) | `+1` via CAS retry |
| `put` overwriting an existing slot | old eTag `−1`, new eTag `+1` |
| `publish` | `+1` (version holds independent reference) |
| `delete` slot | `−1` |
| `deleteVersion` | `−1` |

**Slot metadata updates** (`_blob_slots_*`) also use a CAS retry loop (max 5
attempts) to prevent TOCTOU races between concurrent `put()` / `delete()` calls
on the same record. The retry reads the current entries, re-applies the mutation,
and re-saves with the correct `expectedVersion`.

When `refCount` reaches 0 the blob is **eligible** for deletion. It is not
immediately removed — `vault.blobGC()` performs the actual purge, so callers
retain control over when storage is reclaimed (e.g. skip purge during export
or backup windows).

**Stores without CAS (`casAtomic: false`):** `to-file` and `to-aws-s3` do not
support atomic CAS. On these stores, `refCount` updates are best-effort.
`vault.blobGC()` (v0.13) must do a full reconciliation scan — walk all
`_blob_slots_*` collections and recompute `refCount` from actual references
before purging. This is the safe fallback.

#### Selective versioning

Only explicitly labelled snapshots are retained. No automatic per-write history.

```ts
const set = invoices.blob('inv-001')

// Overwrite slot — no history kept
await set.put('invoice_en', pdfBytes)

// Publish current slot as a named snapshot
await set.publish('invoice_en', 'issued-2025-01')
// → refCount on this eTag +1 (slot ref + version ref = 2)

// Amendment: overwrite slot (old eTag refCount −1)
await set.put('invoice_en', amendedBytes)
await set.publish('invoice_en', 'amendment-2025-02')
// → 'issued-2025-01' version still holds its eTag reference (refCount still > 0)
// → both PDFs are retrievable
```

#### MIME type auto-detection

When `mimeType` is not provided in `put()` options, noy-db performs lightweight
magic-byte detection on the first 16 bytes of the plaintext before compression:

| Magic bytes | Detected type |
|---|---|
| `%PDF` | `application/pdf` |
| `\x89PNG` | `image/png` |
| `\xFF\xD8\xFF` | `image/jpeg` |
| `GIF87a` / `GIF89a` | `image/gif` |
| `PK\x03\x04` | `application/zip` |
| `\x1F\x8B` | `application/gzip` |
| (none matched) | `application/octet-stream` |

This is intentionally minimal — no filename extension guessing, no `file(1)`
equivalent. Callers who need precise MIME types should pass them explicitly.

Auto-detected types also inform the `compress` default: already-compressed
formats (`image/jpeg`, `image/png`, `application/zip`, `application/gzip`)
skip gzip compression automatically unless `compress: true` is explicit.

#### HTTP response surface

`BlobSet.response(slotName, opts?)` returns a native `Response` object for direct
use in any Fetch-API-compatible handler (service workers, Hono, Nitro, H3):

```ts
// Inline preview
const res = await set.response('invoice_en', { inline: true })

// Forced download with custom filename
const res = await set.response('invoice_en', { filename: 'Invoice_001.pdf' })

// Published version
const res = await set.responseVersion('invoice_en', 'issued-2025-01', { inline: true })
```

Response headers:
- `Content-Type` — from `SlotRecord.mimeType`, fallback `application/octet-stream`
- `Content-Length` — original uncompressed size
- `ETag` — quoted HMAC-SHA-256 hex (opaque, stable across re-uploads of same content)
- `Content-Disposition` — `inline; filename="..."` or `attachment; filename="..."`
- `Last-Modified` — RFC 7231 from `SlotRecord.updatedAt`

#### Streaming behaviour (v0.12 vs v0.13)

**v0.12 (current):** `response()` returns a `ReadableStream` body, but the
implementation is **buffered** — all chunks are fetched, decrypted, concatenated,
and decompressed in memory before being enqueued as a single output chunk. This
is functionally correct and the API shape is forward-compatible, but memory usage
is O(blob size). Acceptable for the v0.12 target scale (blobs under ~50 MB).

**v0.13 (streaming decompression):** chunks are piped through a `TransformStream`
that decrypts each chunk individually and feeds them into a `DecompressionStream`.
Each decrypted chunk is enqueued immediately — memory usage is O(chunk size).
For clients that send `Accept-Encoding: gzip`, the decompression step can be
skipped entirely: set `Content-Encoding: gzip` and stream compressed chunks
directly, saving CPU on both sides.

The v0.12 implementation MUST NOT claim "never assembled in memory" in doc
comments. The `ReadableStream` wrapper is present for API compatibility, not
for actual streaming behaviour.

```ts
// Slot management
blob.put(slot, bytes, opts?)        → Promise<void>

blob.delete(slot)                   → Promise<void>

// Published versions (selective history — UC-3)
blob.publish(slot, label)           → Promise<void>

blob.listVersions(slot)             → Promise<VersionRecord[]>
blob.deleteVersion(slot, label)     → Promise<void>
blob.responseVersion(slot, label, opts?) → Promise<Response | null>

// Diagnostics
blob.blobInfo(slot)                 → Promise<BlobObject | null>
```

#### `VersionRecord`

```ts
interface VersionRecord {
  /** User-defined label (e.g. `'issued-2025-01'`, `'amendment-2025-02'`). */
  readonly label: string
  /** eTag of the blob snapshot at publish time — independent of the current slot. */
  readonly eTag: string
  /** ISO timestamp when the version was published. */
  readonly publishedAt: string
  /** User ID of the publisher, if available. */
  readonly publishedBy?: string
}
```

#### Version publish/delete flow

**`publish(slot, label)`:**
1. Read the current `SlotRecord` for the slot → get its `eTag`.
2. Write a `VersionRecord` to `_blob_versions_{collection}/{recordId}/{slotName}/{label}`.
3. CAS-increment `refCount` on the blob's `BlobObject` (`+1`).

The published version holds an independent reference to the blob. Even if the
slot is later overwritten or deleted, the published version's eTag keeps the
blob alive (refCount > 0).

**`deleteVersion(slot, label)`:**
1. Read the `VersionRecord` → get its `eTag`.
2. Delete the `VersionRecord` from `_blob_versions_*`.
3. CAS-decrement `refCount` on the blob's `BlobObject` (`−1`).

**Label uniqueness:** labels are unique per `(recordId, slotName)`. Publishing
with an existing label overwrites the previous version record. If the old and
new eTags differ, the old eTag's refCount is decremented and the new one's is
incremented.

## Encryption invariants

All blob operations respect the zero-knowledge guarantee:

- **Blob data** (`_blob_chunks`): AES-256-GCM, fresh 12-byte random IV per chunk,
  vault-shared `_blob` DEK. Each chunk includes AAD binding (`eTag:index:count`).
- **Blob metadata** (`_blob_slots_*`, `_blob_versions_*`): AES-256-GCM,
  parent collection DEK — same rotation domain as the record it belongs to.
- **Blob index** (`_blob_index`): AES-256-GCM, `_blob` DEK.
- **Keyed eTag**: `HMAC-SHA-256(blobDEK, plaintext)` — opaque to the store.
  Stable across re-uploads of the same content within a vault. NOT stable across
  `_blob` DEK rotations (eTags must be recomputed — see deferred scope).
- **DEK rotation** (`vault.rotate(['invoices'])`): rotates the `invoices` DEK,
  automatically invalidating `_blob_slots_invoices` access for revoked users.
  The `_blob` DEK (chunk data) is a separate rotation target.
- **Stores never see plaintext**: the blob store receives only base64-encoded
  AES-GCM ciphertext in the standard `EncryptedEnvelope` shape.

### Known trade-offs

**Compression oracle (gzip before encryption):** compressing before encrypting
reveals information about plaintext entropy via `compressedSize`. This is the
same class of vulnerability as CRIME/BREACH in TLS. Risk is low in noy-db's
threat model:
- `compressedSize` is stored in `BlobObject`, which is itself AES-GCM encrypted
  under the `_blob` DEK. An attacker without the DEK cannot read it.
- The observable signal is chunk count (via `store.list()` on `_blob_chunks`).
  For 256 KB chunks, this gives ~256 KB granularity — too coarse for a
  practical oracle.
- Already-compressed MIME types (JPEG, PNG, ZIP, GZIP) skip compression
  automatically, eliminating the oracle for the most common file types.

**Deduplication scope:** keyed eTags (HMAC) scope deduplication to a single
vault. Two vaults with identical files produce different eTags and store
separate copies. This is the correct trade-off: cross-vault dedup would require
a shared key that breaks vault isolation.
